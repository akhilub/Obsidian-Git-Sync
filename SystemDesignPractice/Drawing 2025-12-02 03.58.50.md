## Text Elements
The Notification Service provides a simple, unauthenticated way for internal systems to send notifications via email and SMS, with status tracking and a persistent audit trail. The design is intentionally minimal while remaining reliable, observable, and safe for production use. ^GmFtDnjj

• Accept a Notification Request

The service accepts a request containing:
- channel: email or sms
- destination: email address or phone number
- message: direct message OR {templateId, variables}
- callingSource: client/system identifier
- correlationId (optional): if not provided, generate one

• Dispatch Notification

- Emails → SendGrid API
- SMS → Twilio API
- Handle transient failures (retry) vs permanent failures (stop, mark failed)
- Capture provider error codes in audit trail

• Persist an Audit Trail

Immutably store:
- Notification ID (UUID)
- Correlation ID
- Time received
- Channel, destination
- Calling source
- Dispatch attempts (timestamps, error codes, provider response)
- Final status

• Status Lookup

User can call /status/{notificationId} to see:
- accepted
- sending
- sent
- failed (and final provider error)

• Safe Operation Without Authentication

Since the API is public internally, the service applies:
- Input validation (email/phone format, size limits)
- Per-source rate limits (e.g., token bucket per callingSource)
- Quotas (daily message caps)
- WAF-level IP throttling
- Secrets management via AWS Secrets Manager ^eoPENPSU

• Reasons for Choices

- API Gateway + Lambda: minimal ops, native rate limiting, secure.

- AWS Secrets Manager:
    - Safely store SendGrid & Twilio keys. ^4qIs9ZCv

5. Data Model (Minimal + Traceable) ^rSXewqxw

• Request Intake
    1. API Gateway receives request.
    2. Lambda validates:
          - channel in {email/sms}
          - destination format
          - message or template reference
          - size limits (max message length)

    3. Generate notificationId + correlationId if missing
    4. Write RECEIVED event to DynamoDB
    5. Push a message to SQS containing:
         - notificationId
         - channel
         - destination 
         - normalized message

    6. Respond immediately with accepted ^0yAOzgn6

6. Data Flow ^uEpxKxfM

• Dispatcher Flow
    1. Poll message from SQS
    2. Write an ATTEMPT event
    3. Call email provider (SendGrid) or SMS provider (Twilio)
    4. If:
       - 200-level success → write FINAL: sent
       - transient error (5xx, timeouts) → retry via SQS redelivery
       - permanent error (4xx: invalid email/phone) → write FINAL: failed
       - Delete message on success or permanent failure ^pPc3X1ba

• Status Lookup
    1. Lambda reads all events for the notification via PK
    2. Computes final status by last FINAL event
    3. Returns event trace and status summary ^Nm7nq7zO

• Idempotency
    - Clients may retry POST → same message may arrive multiple times
    Solution:
    - Idempotency key = correlationId + callingSource
    - If duplicate correlationId+callingSource appears within 24 hours, return the same notificationId.

• Retries
    - SQS redrive policy (e.g. 3 retries, exponential backoff)
    - Dead-letter queue for persistent failures

• Provider Errors
    - 5xx → transient → retry
    - 4xx → permanent → fail immediately
    - Timeout → retry with backoff

• Durability
    - Every event written to DynamoDB before dispatching ensures no notification is lost. ^RBQChPJu

7. Reliability & Resilience ^YqaSEgwt

Guardrails

1. Strict Validation
    • Reject invalid phone/email
    • Message size caps
    • Template variables restricted to alphanumeric/underscores

2. Rate Limits
    • API Gateway rate limit per IP
    • Additional per-callingSource logical rate limits (token bucket)

3. Quotas
    • Max N notifications/day per callingSource

4. Secrets Handling
    • SendGrid/Twilio keys in Secrets Manager
    • Lambda retrieves via caching

5. Logging & Monitoring
    • CloudWatch alarms on:
        - High provider error rate
        - DLQ growth
        - Over quota attempts ^LbMGRdlv
